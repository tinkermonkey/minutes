# Stage 2 — Live Transcription UI Plan

## UX Reference

**Design file**: `documentation/minutes-app-screens.pen`
**Relevant frame**: **`/record` — Live Transcription** (top-left in the 2×2 grid)

Before building any component in this stage, use the `ux-design-architect` agent to open the design file and inspect the `/record` frame. Key details to extract:
- `RecordButton`: green Flowbite Button with mic icon, size and spacing relative to the status badge
- `SessionStatusBadge`: gray "Ready" badge with dot indicator — position and alignment
- **Warning banner**: amber `Alert` at the top of the content area (speech-swift unreachable state) — the design shows both the normal state and the warning state
- `SegmentCard` layout: speaker chip column width, timestamp column width, transcript text wrapping, card padding and gap
- Speaker chip colors: blue = SPEAKER_00, green = SPEAKER_01, purple = SPEAKER_02, yellow = SPEAKER_03 — use these exact Tailwind color classes
- `NewSpeakerBanner`: amber alert below the top bar, above the transcript panel

Use the `ux-design-architect` agent to re-inspect the design during implementation if spacing or visual hierarchy questions arise.

---

## Goal

A user can open the app, press Record, and watch a live diarized transcript build in real time. The record button is disabled when speech-swift is unreachable. Unknown speakers surface an inline prompt linking to `/speakers`.

---

## What This Stage Adds

- `/record` route: record button, session status indicator, live transcript panel
- TanStack Query hooks wrapping `start_session` / `stop_session` Tauri commands
- `segment_added` event listener → appends segment cards, auto-scrolls
- `new_speaker` event listener → inline banner linking to `/speakers`
- Speaker chip color system (deterministic color per `speaker_id`)
- No new Rust code — this stage is pure frontend wired to the Stage 1 pipeline

## Tasks

### 1. Shared TypeScript types

**`src/types/transcript.ts`**

Define types that mirror the Rust event payloads from `events/mod.rs`:

```ts
export interface Segment {
  id:              number;
  session_id:      number;
  speaker_id:      number;       // non-null in Stage 1/2 events; becomes nullable after Stage 3 migration
  speaker_label:   string;       // from speech-swift (e.g. "SPEAKER_00")
  display_name:    string | null;
  start_ms:        number;
  end_ms:          number;
  transcript_text: string;
}

/**
 * Minimal shape of the new_speaker event payload (SpeakerEvent in Rust).
 * Intentionally 3 fields — it is a notification, not a full data transfer.
 * Stage 3 introduces Speaker (6 fields) and SpeakerWithStats (7 fields) for
 * the speaker registry page. Do not conflate these types.
 */
export interface SpeakerNotification {
  id:              number;
  speech_swift_id: number;
  display_name:    string | null;
}
```

These types are consumed by event listeners and passed as props — no API layer needed in Stage 2.

> **Type naming**: The `new_speaker` event payload is typed as `SpeakerNotification` here to avoid confusion with the richer `Speaker` type introduced in Stage 3. When Stage 3 is built, `src/types/speaker.ts` will define the full `Speaker` type used by the speakers page and commands.

### 2. TanStack Query hooks

**`src/hooks/useSession.ts`**

Wrap the Tauri `start_session` and `stop_session` commands in mutations. Use `useMutation` rather than `useQuery` — these are imperative actions, not data fetches.

```ts
import { useMutation } from '@tanstack/react-query';
import { invoke } from '@tauri-apps/api/core';

export function useStartSession() {
  return useMutation({
    mutationFn: (): Promise<number> => invoke('start_session'),
  });
}

export function useStopSession() {
  return useMutation({
    mutationFn: (sessionId: number): Promise<void> =>
      invoke('stop_session', { sessionId }),
  });
}
```

**`src/hooks/useSpeechSwiftStatus.ts`**

Read the cached health check result from Stage 0:

```ts
import { useQuery } from '@tanstack/react-query';
import { invoke } from '@tauri-apps/api/core';

export function useSpeechSwiftStatus() {
  return useQuery({
    queryKey: ['speech_swift_status'],
    queryFn: (): Promise<boolean> => invoke('get_speech_swift_status'),
    staleTime: Infinity,   // updated only via event, not polling
  });
}
```

The `speech_swift_unreachable` event (from Stage 0) invalidates this query key:

```ts
// in RecordRoute useEffect:
const unlisten = await listen('speech_swift_unreachable', () => {
  queryClient.setQueryData(['speech_swift_status'], false);
});
```

### 3. Session state machine

The record button drives a local state machine in `RecordRoute`. Keep it simple — three states, no external library:

```ts
type SessionState =
  | { status: 'idle' }
  | { status: 'recording'; sessionId: number; startedAt: Date }
  | { status: 'stopping'; sessionId: number };
```

**Transitions**:

```
idle ──[press Record]──→ recording
  (calls start_session, stores sessionId + startedAt)

recording ──[press Stop]──→ stopping
  (calls stop_session; disables button while awaiting response)

stopping ──[stop_session resolves]──→ idle
  (clears segments list for next session)
```

Store in `useState<SessionState>` local to `RecordRoute`. Do not lift to global state — no other route needs it in Stage 2.

### 5. `SessionStatusBadge` component

**`src/components/SessionStatusBadge.tsx`**

Displays alongside the record button. Uses Flowbite `Badge`:

| State | Badge color | Label |
|---|---|---|
| `idle` | gray | "Ready" |
| `recording` | red | "Recording · m:ss" |
| `stopping` | yellow | "Stopping…" |

The `m:ss` elapsed time in recording state is driven by a `setInterval` tick in the parent (`RecordRoute`) that updates every second while `status === 'recording'`. Derive elapsed from `Date.now() - startedAt.getTime()`.

### 6. `segment_added` event listener

In `RecordRoute`, subscribe to the `segment_added` Tauri event using `@tauri-apps/api/event`:

```ts
import { listen } from '@tauri-apps/api/event';

// Inside useEffect (with sessionState.status === 'recording' guard):
const unlisten = await listen<Segment>('segment_added', (event) => {
  setSegments(prev => [...prev, event.payload]);
});
return () => unlisten();
```

**State**:

```ts
const [segments, setSegments] = useState<Segment[]>([]);
```

Clear `segments` when transitioning from `stopping → idle` so the next session starts with an empty panel.

**Performance note**: TanStack Virtual is available (installed in Stage 0) but not required here — transcripts are unlikely to exceed a few hundred segments per session. Use a plain scrollable `div` for now; virtual scrolling can be added in Stage 6 if needed.

### 7. `TranscriptPanel` component

**`src/components/TranscriptPanel.tsx`**

```tsx
interface Props {
  segments: Segment[];
}
```

Renders a vertically scrolling container. Each segment becomes a `SegmentCard`.

**Auto-scroll**: Use a `ref` on a sentinel `div` at the bottom of the list. After each new segment appends, call `bottomRef.current?.scrollIntoView({ behavior: 'smooth' })` in a `useEffect` that depends on `segments.length`.

Only auto-scroll if the user has not manually scrolled up. Detect this with an `onScroll` handler on the container:

```ts
const isAtBottom = container.scrollHeight - container.scrollTop - container.clientHeight < 64;
```

Only call `scrollIntoView` when `isAtBottom` is true. This prevents yanking the view while the user is reviewing earlier content.

**Empty state**: When `segments` is empty and `status === 'recording'`, show a centered Flowbite `Spinner` with label "Waiting for speech…". When `status === 'idle'` and `segments` is empty, show nothing (the record button is the focus).

### 8. `SegmentCard` component

**`src/components/SegmentCard.tsx`**

```tsx
interface Props {
  segment: Segment;
}
```

Layout (horizontal flex row):

```
[SpeakerChip]  [TimestampLabel]  [TranscriptText]
```

- **`SpeakerChip`**: Flowbite `Badge` with a background color derived from `speaker_id` (see §9). Label is `display_name ?? speaker_label`. If `speaker_id` is `null` (possible after a speaker is deleted in Stage 3), render a gray "Unknown" chip using `speakerColor(0)` and label "Unknown".
- **`TimestampLabel`**: `start_ms` formatted as `m:ss` (e.g. `1:23`). Muted text, monospace font (`font-mono`), non-wrapping.
- **`TranscriptText`**: plain text, full width, wraps naturally.

Card itself: white background, subtle border, `rounded-lg`, `p-3`, `mb-2`. No hover state or click action in Stage 2.

> **Forward-compatibility**: Stage 3 makes `segments.speaker_id` nullable (migration recreates the table). Live `segment_added` events in Stage 1/2 always carry a non-null `speaker_id`, so the null case will not arise during recording. However, `SegmentCard` is also reused in Stage 4's `TranscriptReplayPanel`, where historical segments with deleted speakers will have `speaker_id = null`. Implementing the null guard here ensures Stage 4 works correctly without modifying this component later.

### 9. Speaker chip color system

Assign a stable, visually distinct color to each `speaker_id` without storing a mapping — derive it deterministically:

```ts
// src/lib/speakerColor.ts

const CHIP_COLORS = [
  'bg-blue-100   text-blue-800',
  'bg-green-100  text-green-800',
  'bg-purple-100 text-purple-800',
  'bg-yellow-100 text-yellow-800',
  'bg-pink-100   text-pink-800',
  'bg-indigo-100 text-indigo-800',
  'bg-orange-100 text-orange-800',
  'bg-teal-100   text-teal-800',
];

export function speakerColor(speakerId: number): string {
  return CHIP_COLORS[speakerId % CHIP_COLORS.length];
}
```

Apply the returned class string to the Flowbite `Badge` via its `className` prop. This keeps the same speaker visually consistent across all segments in a session and across sessions (same `speaker_id` → same color).

### 10. `NewSpeakerBanner` component

**`src/components/NewSpeakerBanner.tsx`**

In `RecordRoute`, subscribe to the `new_speaker` event:

```ts
const [newSpeakers, setNewSpeakers] = useState<SpeakerNotification[]>([]);

useEffect(() => {
  const unlisten = listen<SpeakerNotification>('new_speaker', (event) => {
    setNewSpeakers(prev => {
      // Deduplicate by speech_swift_id
      const exists = prev.some(s => s.speech_swift_id === event.payload.speech_swift_id);
      return exists ? prev : [...prev, event.payload];
    });
  });
  return () => { unlisten.then(fn => fn()); };
}, []);
```

Render a Flowbite `Alert` (color `warning`) above the transcript panel when `newSpeakers.length > 0`:

```
⚠  Unknown speaker detected — name them in the Speaker Registry →
                                                          [Dismiss]
```

The arrow link uses TanStack Router's `<Link to="/speakers" />`. The Dismiss button calls `setNewSpeakers([])`. Clear `newSpeakers` on `idle → recording` transition as well.

Show one banner regardless of how many unknown speakers were detected (don't stack banners per speaker). The `/speakers` page shows all unnamed speakers.

### 11. Full `RecordRoute` assembly

**`src/routes/record.tsx`**

```tsx
export function RecordRoute() {
  const { data: speechSwiftOk } = useSpeechSwiftStatus();
  const startSession  = useStartSession();
  const stopSession   = useStopSession();
  const queryClient   = useQueryClient();

  const [sessionState, setSessionState] = useState<SessionState>({ status: 'idle' });
  const [segments, setSegments]         = useState<Segment[]>([]);
  const [newSpeakers, setNewSpeakers]   = useState<Speaker[]>([]);
  const [elapsed, setElapsed]           = useState(0);

  // Elapsed timer
  useEffect(() => {
    if (sessionState.status !== 'recording') return;
    const id = setInterval(() => {
      setElapsed(Date.now() - sessionState.startedAt.getTime());
    }, 1000);
    return () => clearInterval(id);
  }, [sessionState]);

  // segment_added listener
  useEffect(() => {
    const p = listen<Segment>('segment_added', e =>
      setSegments(prev => [...prev, e.payload])
    );
    return () => { p.then(fn => fn()); };
  }, []);

  // new_speaker listener
  useEffect(() => { /* see §10 */ }, []);

  // speech_swift_unreachable listener
  useEffect(() => {
    const p = listen('speech_swift_unreachable', () =>
      queryClient.setQueryData(['speech_swift_status'], false)
    );
    return () => { p.then(fn => fn()); };
  }, [queryClient]);

  async function handleRecord() {
    if (sessionState.status === 'idle') {
      const sessionId = await startSession.mutateAsync();
      setSessionState({ status: 'recording', sessionId, startedAt: new Date() });
      setSegments([]);
      setNewSpeakers([]);
      setElapsed(0);
    } else if (sessionState.status === 'recording') {
      const { sessionId } = sessionState;
      setSessionState({ status: 'stopping', sessionId });
      await stopSession.mutateAsync(sessionId);
      setSessionState({ status: 'idle' });
    }
  }

  return (
    <div className="flex flex-col gap-4 p-6 h-full">
      <div className="flex items-center gap-4">
        <RecordButton
          state={sessionState}
          disabled={speechSwiftOk === false}
          onStart={handleRecord}
          onStop={handleRecord}
        />
        <SessionStatusBadge state={sessionState} elapsedMs={elapsed} />
      </div>

      {newSpeakers.length > 0 && (
        <NewSpeakerBanner onDismiss={() => setNewSpeakers([])} />
      )}

      <TranscriptPanel segments={segments} isRecording={sessionState.status === 'recording'} />
    </div>
  );
}
```
