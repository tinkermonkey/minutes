## Goal

Users can type a natural-language query and get ranked transcript excerpts back, optionally filtered by speaker and date range. The embedding infrastructure is already in place from Stage 1 â€” this stage adds the search command and the `/search` UI on top of it.

---

## What This Stage Adds

**Rust**:
- `db::search` module: ANN query via sqlite-vec, JOIN to segments + sessions, optional filter application
- `commands::search`: embeds the query string via fastembed-rs, calls `db::search`, returns ranked results

**Frontend**:
- `/search` route: query input, speaker dropdown, date range filter, result cards
- New types, one mutation hook, and a set of search-specific components

### 2. DB layer â€” `db::search`

**`search_segments`**

sqlite-vec requires the query embedding to be passed as a byte blob (same format as insert: `f32` values in little-endian order). The ANN search runs first via a CTE, then the results are JOINed to segments, sessions, and speakers with optional filter predicates.

```rust
pub fn search_segments(
    conn:      &Connection,
    embedding: &[f32],
    filters:   &SearchFilters,
) -> anyhow::Result<Vec<SearchResult>> {
    let limit = filters.limit.unwrap_or(50).min(100) as i64;

    // Serialize embedding to LE bytes for sqlite-vec
    let blob: Vec<u8> = embedding.iter()
        .flat_map(|f| f.to_le_bytes())
        .collect();

    // Build optional filter predicates
    // Each predicate is nullable-safe: passing NULL skips the condition
    let sql = r#"
        WITH matches AS (
            SELECT segment_id, distance
            FROM segment_embeddings
            WHERE embedding MATCH ?1
            ORDER BY distance
            LIMIT ?2
        )
        SELECT
            sg.id,
            sg.transcript_text,
            sg.start_ms,
            sg.end_ms,
            sg.speaker_id,
            sp.display_name,
            ss.id,
            ss.label,
            ss.created_at,
            m.distance
        FROM matches m
        JOIN  segments sg ON sg.id = m.segment_id
        JOIN  sessions ss ON ss.id = sg.session_id
        LEFT JOIN speakers sp ON sp.speech_swift_id = sg.speaker_id
        WHERE (?3 IS NULL OR sg.speaker_id = ?3)
          AND (?4 IS NULL OR ss.created_at >= ?4)
          AND (?5 IS NULL OR ss.created_at <= ?5)
        ORDER BY m.distance ASC
    "#;

    let mut stmt = conn.prepare(sql)?;
    let results = stmt.query_map(
        rusqlite::params![
            blob,
            limit,
            filters.speaker_id,
            filters.start_date,
            filters.end_date,
        ],
        |row| {
            let distance: f64 = row.get(9)?;
            Ok(SearchResult {
                segment_id:         row.get(0)?,
                transcript_text:    row.get(1)?,
                start_ms:           row.get(2)?,
                end_ms:             row.get(3)?,
                speaker_id:         row.get(4)?,
                display_name:       row.get(5)?,
                session_id:         row.get(6)?,
                session_label:      row.get(7)?,
                session_created_at: row.get(8)?,
                score: (1.0 - distance as f32).max(0.0),
            })
        },
    )?;

    results.collect::<Result<Vec<_>, _>>().map_err(Into::into)
}
```

**Why nullable params for filters**: Passing `None` as `?3` / `?4` / `?5` causes SQLite to evaluate `NULL IS NULL` â†’ true, which short-circuits the filter predicate. This avoids string-building dynamic SQL while keeping one query for all filter combinations.

**Index note**: `segment_embeddings` is a sqlite-vec `vec0` virtual table with a built-in ANN index. No additional index creation is needed. The CTE limit of 50 (or up to 100) keeps the ANN scan fast even with thousands of segments.

### 3. Tauri command â€” `commands::search`

**`src-tauri/src/commands/search.rs`** (new file)

```rust
use crate::{db, embed, state::AppState};

#[tauri::command]
pub async fn search_segments(
    query:   String,
    filters: db::search::SearchFilters,
    state:   tauri::State<'_, AppState>,
) -> Result<Vec<db::search::SearchResult>, String> {
    if query.trim().is_empty() {
        return Ok(vec![]);
    }

    // Embed the query on a blocking thread â€” fastembed is CPU-bound
    let embedding = tokio::task::spawn_blocking(move || embed::embed(&query))
        .await
        .map_err(|e| e.to_string())?
        .map_err(|e| e.to_string())?;

    let db = state.db.lock().unwrap();
    db::search::search_segments(&db, &embedding, &filters)
        .map_err(|e| e.to_string())
}
```

Register in `lib.rs`'s `invoke_handler`.

**`spawn_blocking` rationale**: `embed::embed()` calls ONNX inference synchronously. Running it on the async executor would block the Tauri runtime thread. `spawn_blocking` moves it onto tokio's blocking thread pool, keeping the runtime responsive.

**Empty query guard**: Return early with an empty Vec rather than embedding an empty string, which would return meaningless results.

### 5. Search mutation hook â€” `src/hooks/useSearch.ts`

Use `useMutation` rather than `useQuery`: search is user-triggered, not a background subscription, and results should not be refetched automatically on window focus or mount.

```ts
import { useMutation } from '@tanstack/react-query';
import { invoke } from '@tauri-apps/api/core';
import type { SearchResult, SearchFilters } from '../types/search';

interface SearchParams {
  query:   string;
  filters: SearchFilters;
}

export function useSearch() {
  return useMutation({
    mutationFn: ({ query, filters }: SearchParams): Promise<SearchResult[]> =>
      invoke('search_segments', { query, filters }),
  });
}
```

The mutation's `data`, `isPending`, and `error` fields drive all UI state in the search route.

### 15. Full `SearchRoute` assembly

**`src/routes/search.tsx`**

```tsx
export function SearchRoute() {
  const [query,   setQuery]   = useState('');
  const [filters, setFilters] = useState<SearchFilters>({
    speaker_id: null,
    start_date: null,
    end_date:   null,
  });
  const [hasSearched, setHasSearched] = useState(false);

  const search = useSearch();

  function handleSubmit() {
    if (!query.trim()) return;
    setHasSearched(true);
    search.mutate({ query, filters });
  }

  function updateFilters(patch: Partial<SearchFilters>) {
    setFilters(prev => ({ ...prev, ...patch }));
  }

  return (
    <div className="p-6 flex flex-col gap-4 h-full">
      <h1 className="text-2xl font-semibold">Search</h1>

      <SearchForm
        query={query}
        filters={filters}
        isPending={search.isPending}
        onQueryChange={setQuery}
        onFiltersChange={updateFilters}
        onSubmit={handleSubmit}
      />

      {search.error && (
        <SearchStatus error={search.error.message} />
      )}

      <SearchResultList
        results={search.data ?? []}
        isPending={search.isPending}
        query={query}
        hasSearched={hasSearched}
      />
    </div>
  );
}
```

**Filter change behaviour**: Changing a filter does NOT automatically re-run the search. The user must press Search again (or Enter). This is intentional â€” automatic re-search on every filter change would trigger multiple embeddings. The form state and the last search results are independent: the user can adjust filters, then decide to search.

## Edge Cases and Failure Modes

| Scenario | Behaviour |
|---|---|
| fastembed model not yet loaded | `embed::embed()` will block until the model is ready (loaded in background at Stage 1 startup); the search button spinner covers the wait |
| No segments in DB | sqlite-vec ANN returns 0 rows â†’ empty results, no error |
| All matching segments filtered out | SQL WHERE eliminates ANN matches â†’ empty results message (not an error) |
| speech-swift unreachable | Does not affect search â€” embeddings are already stored in `segment_embeddings`; search works fully offline |
| Query string very long (> 512 tokens) | fastembed truncates to model's max input length (256 tokens for `all-MiniLM-L6-v2`); no crash, quality degrades gracefully |
| sqlite-vec extension not loaded | `search_segments` returns an error; `SearchStatus` shows the error alert |
